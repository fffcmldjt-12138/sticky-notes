# 数据安全、交互稳定性与统一搜索设计

## 目标

在不引入数据库、云同步或完整项目管理能力的前提下，完成三个基础能力：

1. 降低 `notes.json`、`config.json` 和本地附件因异常退出、并发写入或误操作而丢失的风险。
2. 统一主面板、便签小窗和文件夹小窗的编辑状态，避免旧窗口覆盖新内容，并改善输入焦点、中文输入法和长列表体验。
3. 为笔记、待办、子待办、文件夹、标签和附件提供统一的本地搜索与筛选入口。

本轮只预留外部创建接口的内部边界，不开放命令行、Codex Skill、浏览器扩展或思源联动。完整周计划功能不进入本轮。

## 非目标

- 不增加 SQLite、全文搜索数据库或远程服务。
- 不实现云备份、跨设备同步或双向同步。
- 不实现完整操作历史、版本树或多人协作冲突解决。
- 不实现小时级日历、周计划、习惯追踪或 AI 自动规划。
- 不实现 PDF 正文索引、图片 OCR 或语义搜索。
- 不允许渲染进程直接读写本地数据文件。

## 总体架构

主进程继续作为数据和系统能力的唯一所有者。新增或收敛为以下边界：

- `SafeJsonStore`：负责串行、原子式 JSON 保存、结构校验和恢复。
- `BackupService`：负责变更快照、每日备份、清理和恢复前保护。
- `NoteStore`：继续提供领域操作，并给项目与文件夹维护修订版本。
- `ChangeBroadcaster`：在领域操作成功后向所有窗口广播规范化变更。
- `SearchService`：从当前内存数据构建轻量索引并执行查询。
- `ExternalCaptureService`：只定义内部创建请求入口，当前没有外部传输层。

渲染进程只维护界面草稿和当前数据快照，通过 preload IPC 调用主进程。任何保存、备份、恢复、搜索和导入导出都由主进程完成。

## 一、数据安全

### 1. 串行和原子式保存

每个数据文件拥有独立写入队列。对同一文件的写入严格按提交顺序执行，后发请求不能与前一写入并行覆盖。

保存步骤：

1. 将完整数据序列化到同目录临时文件。
2. 重新读取并解析临时文件，执行版本和基础结构校验。
3. 将当前正式文件复制为本次变更快照。
4. 使用文件替换操作将临时文件变为正式文件。
5. 替换成功后再向调用方返回结果并广播变更。

临时文件名包含进程 ID 和随机值。应用启动时清理超过 24 小时的遗留临时文件，不把临时文件作为自动恢复来源。

### 2. 备份策略

备份目录位于应用用户数据目录下的 `backups`，不混入附件目录。

- 变更快照：保留最近 20 份有效 `notes.json` 快照。
- 每日备份：每天第一次成功保存后生成一份，保留最近 30 个自然日。
- `config.json`：保留最近 5 份变更快照，不做每日备份。
- 恢复前保护：每次恢复或导入前，无条件保存当前数据副本，不计入普通保留数量，保留 7 天。

快照写入失败不阻止主文件保存，但必须记录诊断日志并在设置页显示非阻塞警告。主文件替换失败则本次领域操作视为失败，不广播成功状态。

### 3. 启动校验和自动恢复

启动时依次执行：

1. 读取正式文件并完成 JSON、版本和领域结构校验。
2. 正式文件有效时正常启动。
3. 正式文件无效时，从新到旧检查变更快照和每日备份。
4. 找到最近有效备份后，先保留损坏文件，再恢复备份，并记录恢复原因。
5. 所有备份均无效时，以空数据启动只读故障页，不自动覆盖损坏文件。

自动恢复后必须在界面显示一次明确通知，包含恢复来源、备份时间和“打开数据目录”入口。

### 4. 导入、导出与恢复

设置页增加“数据与备份”区域：

- 打开数据目录。
- 立即创建备份。
- 查看备份列表。
- 从选中备份恢复。
- 导出完整数据包。
- 导入完整数据包。
- 查看附件占用并清理未使用附件。

完整数据包使用 ZIP，包含带版本的 JSON、附件目录和清单文件。导入时先解压到临时目录、拒绝绝对路径、路径穿越和符号链接，再校验全部内容、显示项目数量和附件数量，由用户确认替换。第一版导入只支持完整替换，不做合并导入。

恢复和导入完成后，主进程重新加载存储、重建搜索索引，并向全部窗口发送一次全量刷新事件。

### 5. 附件清理

附件清理以当前项目中的 `asset://local/...` 引用为准，扫描笔记正文和附件字段。只有满足以下条件的文件才可清理：

- 当前数据中没有引用。
- 不属于回收站中尚可恢复的内容。
- 文件最后修改时间超过 7 天。

扫描结果先展示文件数量和总大小，必须由用户确认后删除。备份包中的附件不受清理操作影响。

## 二、交互稳定性

### 1. 修订版本和写入冲突

`BaseItem` 和 `FolderItem` 增加整数 `revision`，新建时为 1。每次成功领域变更后递增。旧数据迁移时缺失值补为 1。

`NotesFile.version` 从 4 升为 5。迁移只补充修订版本，不改变现有项目 ID、排序、文件夹关系、提醒或附件引用。

渲染进程提交更新时携带自己最后看到的 `expectedRevision`：

- 与主进程当前版本一致：应用更新并返回新版本。
- 不一致：主进程拒绝覆盖，返回当前最新对象和冲突类型。

普通属性冲突不进行字段级自动合并。界面保留本地草稿并提示“内容已在其他窗口更新”，用户可以选择载入新版本或复制本地草稿。这样优先避免静默丢失，而不是假装自动合并成功。

待办完成、移动、置顶等离散命令不提交整个对象，而是继续使用专用领域命令，以减少冲突范围。

### 2. 编辑草稿与持久化状态

每个编辑器区分三种状态：

- `remoteSnapshot`：最近一次主进程确认的数据。
- `localDraft`：用户当前输入。
- `saveState`：空闲、保存中、已保存、冲突、失败。

远端广播到达时：

- 当前没有未保存草稿：更新快照和编辑器。
- 当前正在输入或存在未保存草稿：只更新远端快照，不重建编辑器 DOM。
- 保存成功且版本匹配：以主进程返回值确认草稿。

标题、Markdown 正文和待办输入采用独立草稿，不因其他字段更新而整体重挂载。

### 3. 中文输入法和焦点

编辑控件统一处理 `compositionstart`、`compositionend`：

- 组合输入期间不触发持久化、不接受远端内容替换。
- `compositionend` 后合并最终文本并启动保存防抖。
- 新建待办先完成本地行创建和聚焦，再异步持久化，避免初次点击后短暂无法输入。

组件不得使用随内容变化的 React `key`。保存、排序、提醒更新和标签更新不得主动调用编辑器 `focus()`，除非用户刚执行了明确的新增或打开命令。

### 4. 保存反馈和失败处理

“自动保存”不再作为固定文案。只有最近一次保存成功时显示“已保存”；保存中、失败和冲突必须有不同状态。

保存失败时：

- 本地草稿保留。
- 不回退输入框内容。
- 提供重试和复制草稿操作。
- 写入诊断日志，但不记录正文内容。

### 5. 撤销范围

本轮提供最近操作撤销，不实现无限历史。主进程维护最多 20 条可逆领域命令，应用退出后清空。

支持：

- 删除笔记、待办或文件夹。
- 完成或恢复待办。
- 移动项目或文件夹。
- 移出文件夹。
- 置顶状态变化。

正文编辑继续由编辑器自身撤销栈处理。恢复备份、导入数据、清理附件不可通过普通撤销恢复，执行前使用独立确认和恢复前保护副本。

### 6. 渲染和长列表

先通过稳定属性、组件拆分、记忆化和按项目 ID 更新列表，减少无关卡片重渲染。只有在性能测试证明大量卡片仍有明显问题时才引入虚拟列表，避免虚拟化破坏现有拖拽、文件夹嵌套和编辑焦点。

性能验收数据集为 500 个项目、1000 条待办任务和三层文件夹；搜索输入、列表滚动和普通勾选不应出现持续阻塞。

## 三、统一搜索

### 1. 索引范围

`SearchService` 在主进程内存中维护规范化文档，不创建数据库文件。索引对象包括：

- 笔记：标题、Markdown 纯文本、标签、文件夹路径和附件名。
- 待办：便签标题、任务、子待办、标签、完成状态、四象限、时间和文件夹路径。
- 文件夹：名称和完整层级路径。
- 回收站：默认不进入普通结果，只在用户开启“包含已删除”时搜索。

每次领域变更只重建受影响项目；导入、恢复和文件夹层级大规模变化时执行全量重建。

### 2. 查询规则

第一版使用确定性的本地匹配：

- Unicode 小写规范化。
- 忽略连续空白差异。
- 查询按空格拆为多个关键词，所有关键词必须命中同一结果对象。
- 标题命中优先于标签，标签优先于正文。
- 同级结果按置顶、待办优先级、更新时间排序。

不做拼音、模糊纠错、分词服务或语义向量。搜索结果显示命中字段和有限长度片段，不渲染完整 Markdown。

### 3. 筛选条件

搜索与筛选共用同一查询接口，支持：

- 类型：笔记、待办、文件夹。
- 状态：未完成、已完成、已删除。
- 时间：今天、已逾期、未来 7 天、最近编辑。
- 优先级：重要紧急及其他四象限组合。
- 置顶。
- 标签。
- 所在文件夹。

多个不同类别筛选使用 AND；同一类别的多个值使用 OR。时间筛选基于本机时区计算自然日边界。

### 4. 搜索界面

主面板顶部增加搜索按钮。激活后替换标题区域为搜索输入，不新开页面。结果列表暂时替换普通树形列表，退出搜索后恢复原滚动位置和文件夹展开状态。

点击结果：

- 笔记：打开对应编辑器并保留搜索词提示；第一版不改变正文光标位置。
- 待办或子待办：打开待办编辑器、展开对应任务并聚焦该行。
- 文件夹：退出搜索并滚动到对应文件夹。

无查询文本时显示智能筛选入口：今天、已逾期、未来 7 天、重要紧急、最近编辑。

## 四、外部创建接口预留

新增内部 `ExternalCaptureRequest`：

- 类型：笔记或待办。
- 标题和正文。
- 标签。
- 可选目标文件夹。
- 可选附件导入路径。
- 来源标识和外部请求 ID。

`ExternalCaptureService` 负责校验请求并调用现有领域创建命令。外部请求 ID 用于防止重试时重复创建。

本轮不增加命名管道、本地 HTTP 服务、CLI、URL Scheme 或 Codex Skill，也不在设置页暴露相关开关。该接口仅用于保证未来扩展不会绕过数据安全和广播机制。

## 五、周计划边界

本轮不实现周计划。未来若加入，采用默认关闭的轻量“本周”视图，并复用统一搜索、现有待办和导出能力：

- 不新增第二套任务。
- 不加入小时级日历。
- 不默认占用主面板入口。
- 不做强制打卡或 AI 评价。

是否进入后续版本，取决于本轮搜索和稳定性完成后的真实使用反馈。

## 六、错误处理和诊断

诊断日志只记录时间、操作类型、对象 ID、修订版本、文件路径类别和错误栈，不记录笔记正文、待办文本、标签值或附件内容。

设置页可导出诊断包，包含脱敏日志、应用版本、操作系统版本、数据文件结构摘要和备份清单，不包含用户内容。所有网络功能仍保持关闭。

## 七、测试与验收

### 数据安全

- 并发保存按顺序落盘，最终内容与最后一次成功提交一致。
- 临时文件损坏不会替换正式文件。
- 正式文件损坏时恢复最近有效备份并保留损坏原件。
- 导入前校验失败不会修改现有数据。
- 恢复和导入后所有窗口刷新，搜索索引与数据一致。
- 附件清理不会删除回收站或 7 天保护期内的资源。

### 交互稳定性

- 两个窗口编辑同一项目时，旧修订不能静默覆盖新修订。
- 中文组合输入期间远端广播不改变输入框和光标。
- 新建待办后可以立即输入。
- 保存失败保留草稿，并能重试或复制。
- 离散命令只更新目标项目，不重建无关编辑器。
- 500 个项目压力数据下完成基本性能回归。

### 搜索

- 笔记、任务、子待办、标签、文件夹路径和附件名均可命中。
- 多关键词、筛选组合、时区边界和排序规则具有确定结果。
- 回收站默认排除，开启后可查询。
- 点击结果能打开正确对象；待办结果能展开到具体任务或子待办。
- 退出搜索恢复原列表状态。

### 发布门槛

- 全量单元和组件测试通过。
- Node 与 renderer 类型检查通过。
- 生产构建、安装包构建和打包版冒烟启动通过。
- 使用旧版 `notes.json` 验证迁移、备份和回滚路径。

## 实施顺序

1. 先建立 `SafeJsonStore`、备份格式、校验和恢复测试。
2. 将 `NoteStore`、配置和附件维护接入安全保存。
3. 增加修订版本、冲突响应和统一窗口广播。
4. 收敛编辑草稿、中文输入法、保存反馈和撤销命令。
5. 建立 `SearchService`、筛选规则和搜索界面。
6. 增加设置页数据工具、诊断包和完整导入导出。
7. 定义但不开放 `ExternalCaptureService`。
8. 完成迁移、压力、打包和恢复演练。
