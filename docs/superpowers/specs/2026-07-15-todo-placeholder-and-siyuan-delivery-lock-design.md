# 待办占位任务与思源投送锁设计

## 目标

本次改动解决两个独立但范围较小的问题：

1. 删除待办中的最后一条任务后，保留一条能够立即编辑和保存的真实空任务，避免出现只有界面状态、没有持久化实体的“幽灵任务”。
2. 允许用户对不需要进入思源的笔记设置“禁止投送”，并在所有投送入口和主进程服务中执行该限制。

不改变待办的提醒、重复、四象限和子待办规则，也不增加双向同步或云同步。

## 根因

`TodoEditor` 在收到空任务列表后通过乐观状态创建占位任务，再异步调用 `addTodoTask`。当最后一条任务被删除或用户再次操作时，界面可能保留一个尚未进入 `notes.json`、或已经从数据层删除的待处理任务。随后对这个任务的编辑会携带数据层不存在的任务 ID，因此界面仍可见，但内容无法正常保存。

“至少保留一条任务”属于数据不变量，不应仅由渲染进程维持。

## 待办删除设计

### 数据层不变量

`NoteStore.deleteTodoTask` 删除目标任务后：

- 如果仍有任务，保存正常的过滤结果。
- 如果没有任务，在同一次 `changeTodo` 写入中创建一条新的标准空任务。
- 返回的 `TodoItem` 必须已经包含该空任务，任务 ID 是正式的 `task_*` ID。

该操作只产生一次持久化事务和一次待办 revision 更新，不暴露短暂的空任务列表。

### 渲染进程

删除成功后继续使用返回的完整 `TodoItem` 更新状态。现有任务 ID 变化检测负责聚焦新任务输入框。

`TodoEditor` 仍可为旧数据或异常空列表提供一次兼容性补全，但正常删除流程不再依赖乐观占位任务。待处理任务状态必须在数据层返回正式任务后清理，不能长期显示已不存在的任务。

## 禁止投送设计

### 数据模型

`NoteItem` 新增：

```ts
siyuanDeliveryDisabled: boolean
```

- 新建笔记默认值为 `false`。
- 旧数据迁移和导入校验缺少该字段时补为 `false`。
- 字段通过现有笔记更新 IPC 和 `StickyItemPatch` 持久化。
- 待办不增加该字段，因为当前思源投送只适用于笔记。

### 右键菜单

笔记右键菜单新增切换操作：

- 未禁止时显示“禁止投送到思源”。
- 已禁止时显示“允许投送到思源”。
- 待办菜单不显示该操作。

主面板和独立文件夹窗口必须处理同一 `CardAction`，避免入口行为不一致。

### 标题按钮

被禁止投送的笔记仍保留标题旁按钮：

- 使用灰色锁定图标。
- 按钮不可点击。
- `aria-label` 和悬浮提示为“已禁止投送到思源”。
- 之前的 `siyuanDelivery` 元数据继续保留，不因禁止投送而删除。

解除禁止后，按钮根据现有规则恢复为“发送”“再次发送”或“已发送”。

### 主进程防护

`SiyuanDeliveryService` 在读取目标笔记后、生成文档或调用思源 API 前检查 `siyuanDeliveryDisabled`。若为 `true`，立即抛出明确错误“该笔记已禁止投送到思源”。

这样即使发生旧窗口、竞态或直接 IPC 调用，也不会绕过限制。

## 错误与并发

- 禁止状态使用现有 revision 乐观并发更新，不新增旁路配置文件。
- 更新冲突沿用现有冲突处理，不静默覆盖其他窗口的新状态。
- 删除最后一条任务与创建替代任务必须是同一数据层事务，不能拆成两个 IPC 请求。
- 投送按钮发送过程中若笔记被另一窗口设为禁止，主进程检查负责拒绝本次请求，界面显示现有失败反馈。

## 测试

至少覆盖：

1. `NoteStore` 删除唯一任务后返回并保存一条新的可编辑空任务。
2. 删除多条任务中的一条不会额外创建空任务。
3. `TodoEditor` 接收删除结果后显示并聚焦正式的新任务，输入能够提交正式任务 ID。
4. 新建和迁移后的笔记具有正确的 `siyuanDeliveryDisabled` 默认值。
5. 右键菜单只为笔记显示禁止/允许投送操作。
6. 被禁止笔记显示灰色锁定按钮，按钮不可用；解除后恢复现有状态。
7. `SiyuanDeliveryService` 在任何网络调用前拒绝被禁止笔记。
8. 主面板和独立文件夹窗口均能切换禁止状态。
